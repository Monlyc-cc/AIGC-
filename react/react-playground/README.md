# 所以我们需要一个编译器
1. 将 tsx 编译为 jsx
  - @babel/standalone (babal的浏览器版本) ： 模拟浏览器环境，将js代码编译为适应浏览器的版本
  - @types/babel__standalone ： 为 @babel/standalone 提供 ts 类型定义
   -  前情提要：react 本身就是用 ts 写的，自身存在许多 ts 的类型声明 
   - ts的类型声明？： 类比与js模仿强类型语言而诞生的 ‘类型概念’ ，ts 并不是一门新的语言，而是js的 pro 版本 ，改变传统js的弱类型编译方式，在编译之前增加了一层静态的类型检查，让js语言在编译前先被 进行一次严谨的类型检查，在那之后再被编译为传统的js语言并执行。
   - ts 的优点：
      1. 兼容性强， 可以在ts环境直接编写js语法的代码
      2. 在代码编写时就判断变量的类型是否合法,使得代码的逻辑更加严谨,防范于未然

# 制作一个在线react-playground
主要目标： 
1. 获取在线代码，获取用户写在网页端的代码，并将执行结果展现给用户
2. 在线获取的代码该如何执行？；
  - 两种选择：
    1. 选择1： 将获取的代码直接展现在 本react项目（单页面） 的子组件中
        - 优点：不用对代码进行任何额外的预处理，代码会随着react项目的执行过程被 执行
        - 缺点：react项目是单页面的，单页面!! 意味着只有一份html在执行，如果用户写的代码有错误，会导致整个react项目崩溃，那么这个在线的网站就崩溃了
    2. 选择2 ：（更加安全且有效的选择） 将获取的代码直接插入到另一份完整的新的 html 中执行
        - 优点：保持react项目 （即在线平台）的独立性，不会被用户写的代码影响，即使用户写的代码是错误的，崩溃的只会是另一个html页面，而不是我们的在线平台
        - 缺点： 
            1. 缺少基本的外部资源（react源代码）： 一份新的，完整的html过于原始，而我们项目的需要（在线的react-playground）需要有大量的有关react原始代码，需要做额外的必须引入
            2. 用户便捷、自主的引入外部资源，只能通过在线的url来引入外部的包，处理过程过于繁琐，且不确定性强

3. 依据以上分析，我们选择第二种方法来实现项目的核心功能
